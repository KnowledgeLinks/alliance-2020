First & Second Quarters - Build RDF Publishing Pipeline

We want to add an option to the current Goldrush service, and option for libraries to select a subset of their holdings and be able to publish their MARC holdings as RDF to the web to be indexed by Google, Microsoft Bing, and DuckDuckGo web search engines.

The BIBFRAME RDF is stored in Postgres database that is transformed into HTML RDFa using the Schema.org ontology that is cached on the filesystem along with the sitemaps of the Alliance member libraries.

Beside the RDf web publishing pipeline for the libraries MARC records, we will also provide the option for libraries to receive a download of files in a variety of RDF serializations like XML, n-triples, or Turtle. Using the BIBFRAME ontology, the libraries Works will be de-duplicated with other BIBFRAME Works in the Alliance's collection with their specific BIBFRAME Instances.

Technology

Java RDF middleware runs the individual MARC21 records to MARC XML that is then transformed into the BIBFRAME RDF and stored in a Postgres database, possibly using the Trellis as an API layer. This would replace the existing Alliance BIBCAT implementation of using the Blazegraph triplestore technology that has been effectively abandoned by it's original developers (although the main developers are now employed by Amazon and with the Blazegraph stack used as the basis for Amazon's Neptune graph database service.

Proposed Workflow

  1. Library staff, using the Goldrush UI, searches for MARC records that want converted to BIBFRAME
  2. Backend process is kicked off:
    1. First checks for existing BIBFRAME Instances and Works associated with the Goldrush match key
    2. If BIBFRAME Instance(s) or Work(s) are not found, the MARC record is serialized as XML, run through the Library of Congress marc2bibframe XSLT to produce BIBFRAME RDF. The RDF is then stored in Postgres
    3. The BIBFRAME RDF is transformed into Schema JSON-LD and an HTML5 representation is then cached.
    4. The Alliance sitemap is updated to reflect the changes.
  3. An email is sent to the library staff member notifying them of their newly published RDF

Third & Forth Quarter - Hosted Sinopia Services

The Sinopia linked data editor will be finishing up it's third development work-cycle at the end of second quarter to the beginning of the third quarter of 2020. With the new Alliance RDF publishing services that will be developed in first and second quarters, we should advocate that the Alliance's Goldrush BIBCAT service as an authority source for users to search and link to Alliance resources in the creation of linked data resources in Sinopia.

Technology

Part of the expected Sinopia's third work-cycle deliverables is an attempt to make the supporting infrastructure for Sinopia more cloud agnostic instead of being so tightly coupled to Amazon Web Services like Cognito and Fargate. Given a successful launch of RDF publishing pipeline for the Alliance membership, a hosted version of Sinopia by the Alliance would allow library staff to edit the converted RDF resources and also allow the membership to create new RDF resources.

The current iteration of Sinopia uses Docker containers with the following structure:

  • Node.js server for delivery of a React application using Redux for a rich client-side Javascript RDF editor.
  • The storage-layer for Sinopia uses Trellis, an open-source linked-data platform with a Postgres relational database backend.
  • A Node.js middleware applications for managing Trellis group permissions and for indexing the resources in Trellis into a Elasticsearch search index.
